# CLAUDE.md Template

Canonical section structure for generated CLAUDE.md files. Sections appear in this order. Each section includes its purpose, style rules, and a filled example.

---

## Full Template

```md
# [Project Name]

[One-sentence description of what the project does and who it is for.]

## Tech Stack

- **Language / Runtime:** [e.g., TypeScript / Node.js 20, Python 3.12]
- **Framework:** [e.g., Next.js 14, FastAPI, Express]
- **Database:** [e.g., PostgreSQL via Prisma, SQLite]
- **Key dependencies:** [3–5 notable libs — not an exhaustive list]
- **Package manager:** [npm / pnpm / yarn / pip / cargo / go modules]

## Commands

[Exact runnable commands. No paraphrasing. Copy-paste ready.]

\`\`\`bash
# Development
npm run dev          # starts local dev server at localhost:3000

# Build
npm run build        # production build to dist/

# Tests
npm test             # unit tests
npm run test:e2e     # end-to-end (requires running dev server)

# Lint / Format
npm run lint
npm run format
\`\`\`

## Architecture

[Top-level folder map. One line per directory. Only directories meaningful for navigation.]

\`\`\`
src/
  app/          — Next.js app router pages and layouts
  components/   — shared React components
  lib/          — server utilities, API clients, helpers
  db/           — Prisma schema and migrations
tests/          — unit and integration tests (mirrors src/ structure)
.github/        — CI/CD workflows
\`\`\`

## Code Conventions

[Rules enforced by linter/formatter + any team conventions from grill-me.]

- Single quotes, no semicolons (Prettier config)
- Strict TypeScript — no `any`, no `@ts-ignore` without an explanatory comment
- Named exports only — no default exports in `lib/` or `components/`
- [Additional rules from grill-me answers]

## Testing

[How to run tests, where they live, coverage requirements.]

- Test files live in `tests/` and mirror the `src/` directory structure
- Run unit tests: `npm test`
- Run e2e tests: `npm run test:e2e` (requires dev server on port 3000)
- Coverage gate: 80% — CI blocks merge if below

## What NOT To Do

[Hard rules. Minimum 2. Written as imperatives.]

- Never manually edit files in `dist/` or `.next/` — they are generated by the build
- Never push directly to `main` — all changes require a PR with a passing CI build
- [Additional rules from scan signals or grill-me]

## Environment Setup

[Required before the project runs locally.]

Copy `.env.example` to `.env.local` and fill in:

\`\`\`
DATABASE_URL=          # PostgreSQL connection string (ask team for local value)
NEXT_PUBLIC_API_URL=   # http://localhost:3001 locally
STRIPE_SECRET_KEY=     # Required for payment flows — get from 1Password
\`\`\`

[Any additional one-time setup: `docker compose up -d`, `npm run db:seed`, etc.]
```

## Section-by-Section Style Rules

### Project Overview (opening paragraph)

- One sentence maximum
- Format: "[Project name] is a [what it does] for [who uses it]"
- Must be project-specific — never generic ("This is a web application")
- Do not use a heading for this — it sits directly below the H1

### Tech Stack

- Bullet list, not a table
- Limit to 5 bullets maximum — this is a reference, not a catalog
- Include version numbers when they matter for Claude's decisions (Node 20 vs 18 behavior differs)
- If two frameworks are present (e.g., Next.js + tRPC), list both under Framework

### Commands

- Always use a `bash` code fence
- One command per line
- Use inline comments (`# description`) for context
- Group by category with a blank line between groups: Development, Build, Tests, Lint/Format, Database, Deploy
- Commands come from signal files only — never invented
- If the project uses `make` or `just`, list those wrappers, not the underlying npm/python commands

### Architecture

- Always use a plain code fence (no language tag) for the directory tree
- Only list directories, not individual files
- Maximum depth: 2 levels (top-level dir → immediate subdirectories)
- One-line description per directory, separated by `—` (em dash)
- Omit: `node_modules/`, `.git/`, `dist/`, `build/`, `.next/`, `__pycache__/`, `venv/`

### Code Conventions

- Bullet list
- Each bullet is one enforceable rule
- Derive from lint config first — only add grill-me answers for rules the linter doesn't cover
- Avoid vague entries ("write clean code") — every rule must be actionable

### Testing

- Two to four sentences or bullets — not a wall of text
- Always include: where test files live, how to run tests
- Include coverage threshold if found in config
- If multiple test types exist (unit, integration, e2e), list the command for each

### What NOT To Do

- Imperative bullets: "Never...", "Do not...", "Always use X, not Y"
- Minimum 2, maximum 6 — more than 6 means the project has a documentation problem
- Most valuable entries come from: generated directories (from .gitignore), branch protection (from workflows), and grill-me answer #2
- Do not repeat things already obvious from Code Conventions

### Environment Setup

- Only include if the project has a .env.example or requires services to run
- List every variable name from .env.example — include the comment as the description
- Mark required variables explicitly ("ask team", "get from 1Password", etc.)
- If docker-compose is present, include `docker compose up -d` as the first setup step

## Omission Rules

| Section | Omit when |
|---|---|
| Tech Stack | Never |
| Commands | Never — ask via grill-me if signals are absent |
| Architecture | Project is a single-file script or has no meaningful directory structure |
| Code Conventions | No lint config found and grill-me provided nothing |
| Testing | No test framework detected and no test files found |
| What NOT To Do | Never — ask via grill-me if signals are absent |
| Environment Setup | No .env.example found and no services detected |

---

## Target Length

60–150 lines. Shorter is better. CLAUDE.md is a reference file Claude reads at the start of every session — not documentation for humans. Dense and accurate beats thorough and padded.
